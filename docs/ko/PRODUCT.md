# Paperclip - 제품 정의

## 개요

Paperclip은 자율 AI 회사를 위한 컨트롤 플레인이다. 하나의 Paperclip 인스턴스로 여러 회사를 운영할 수 있다. **회사(company)**는 일급 객체이다.

## 핵심 개념

### 회사 (Company)

회사는 다음을 갖는다:

- **목표(goal)** - 존재 이유 ("3개월 이내에 MRR $1M을 달성하는 #1 AI 노트 앱 만들기")
- **직원(employees)** - 모든 직원은 AI 에이전트
- **조직 구조(org structure)** - 누가 누구에게 보고하는지
- **수익과 비용(revenue & expenses)** - 회사 수준에서 추적
- **태스크 계층(task hierarchy)** - 모든 업무는 회사 목표로 거슬러 올라감

### 직원과 에이전트 (Employees & Agents)

모든 직원은 에이전트이다. 회사를 만들 때, CEO를 먼저 정의하고 그로부터 확장해 나간다.

각 직원은 다음을 갖는다:

- **어댑터 타입 + 설정(adapter type + config)** - 이 에이전트가 어떻게 실행되고 정체성/행동이 무엇인지를 정의한다. 이는 어댑터별로 다르다 (예: OpenClaw 에이전트는 SOUL.md와 HEARTBEAT.md 파일을 사용하고, Claude Code 에이전트는 CLAUDE.md를 사용하며, 순수 스크립트는 CLI 인자를 사용할 수 있다). Paperclip은 형식을 강제하지 않는다 - 어댑터가 결정한다.
- **역할과 보고 체계(role & reporting)** - 직함, 누구에게 보고하는지, 누가 자신에게 보고하는지
- **역량 설명(capabilities description)** - 이 에이전트가 무엇을 하고 언제 관련되는지에 대한 짧은 설명 (다른 에이전트가 누가 무엇을 도울 수 있는지 발견하는 데 도움됨)

예시: CEO 에이전트의 어댑터 설정은 각 하트비트마다 "임원들이 하고 있는 일을 검토하고, 회사 지표를 확인하고, 필요하면 우선순위를 재조정하고, 새로운 전략적 이니셔티브를 배정하라"고 지시한다. 엔지니어의 설정은 "배정된 태스크를 확인하고, 가장 높은 우선순위를 선택해서 작업하라"고 지시한다.

그런 다음 CEO에게 보고하는 사람을 정의한다: 프로그래머를 관리하는 CTO, 마케팅 팀을 관리하는 CMO 등. 트리의 모든 에이전트는 자체 어댑터 설정을 갖는다.

### 에이전트 실행 (Agent Execution)

에이전트의 하트비트를 실행하는 두 가지 기본 모드가 있다:

1. **명령 실행(run a command)** - Paperclip이 프로세스(셸 명령, Python 스크립트 등)를 시작하고 추적한다. 하트비트는 "이것을 실행하고 모니터링하라"이다.
2. **요청 후 잊기(fire and forget a request)** - Paperclip이 외부에서 실행 중인 에이전트에 webhook/API 호출을 보낸다. 하트비트는 "이 에이전트에게 깨어나라고 알린다"이다. (OpenClaw 훅이 이렇게 작동한다.)

합리적인 기본값을 제공한다 - 설정에 따라 Claude Code나 Codex로 셸 아웃하고, 세션 ID를 기억하고, 기본 스크립트를 실행하는 기본 에이전트. 하지만 무엇이든 연결할 수 있다.

### 태스크 관리 (Task Management)

태스크 관리는 계층적이다. 모든 업무는 어느 시점에서든 상위 태스크 체인을 통해 회사의 최상위 목표까지 거슬러 올라갈 수 있어야 한다:

```
I am researching the Facebook ads Granola uses (current task)
  because → I need to create Facebook ads for our software (parent)
    because → I need to grow new signups by 100 users (parent)
      because → I need to get revenue to $2,000 this week (parent)
        because → ...
          because → We're building the #1 AI note-taking app to $1M MRR in 3 months
```

태스크에는 부모가 있다. 모든 태스크는 상위 태스크를 위해 존재하며, 이는 회사 목표까지 이어진다. 이것이 자율 에이전트의 정렬을 유지하는 핵심이다 - 에이전트는 항상 "왜 이 일을 하고 있는가?"에 답할 수 있다.

더 상세한 태스크 구조는 추후 결정(TBD).

## 원칙

1. **에이전트 실행 방식에 무관하다.** 에이전트는 OpenClaw 봇, Python 스크립트, Node 스크립트, Claude Code 세션, Codex 인스턴스 등 무엇이든 될 수 있다. Paperclip은 통신을 위한 컨트롤 플레인을 정의하고 하트비트를 위한 유틸리티 인프라를 제공한다. 에이전트 런타임을 강제하지 않는다.

2. **회사가 조직의 단위이다.** 모든 것은 회사 아래에 존재한다. 하나의 Paperclip 인스턴스, 여러 회사.

3. **어댑터 설정이 에이전트를 정의한다.** 모든 에이전트는 정체성과 행동을 제어하는 어댑터 타입과 설정을 갖는다. 최소 계약은 "호출 가능할 것"이다.

4. **모든 업무는 목표로 이어진다.** 계층적 태스크 관리는 어떤 것도 고립되어 존재하지 않음을 의미한다. 태스크가 회사 목표에 왜 중요한지 설명할 수 없다면, 그것은 존재해서는 안 된다.

5. **컨트롤 플레인이지, 실행 플레인이 아니다.** Paperclip은 오케스트레이션한다. 에이전트는 어디에서든 실행되고 상태를 보고한다.

## 사용자 흐름 (이상적인 시나리오)

1. Paperclip을 열고, 새 회사를 만든다
2. 회사의 목표를 정의한다: "3개월 내 MRR $1M 달성하는 #1 AI 노트 앱 만들기"
3. CEO를 만든다
   - 어댑터를 선택한다 (예: Claude Code용 process 어댑터, OpenClaw용 HTTP 어댑터)
   - 어댑터를 설정한다 (에이전트 정체성, 루프 동작, 실행 설정)
   - CEO가 전략적 분해를 제안 → 이사회가 승인
4. CEO의 직속 보고자를 정의한다: CTO, CMO, CFO 등
   - 각자 자체 어댑터 설정과 역할 정의를 갖는다
5. 그들의 보고자를 정의한다: CTO 아래 엔지니어, CMO 아래 마케터 등
6. 예산을 설정하고, 초기 전략적 태스크를 정의한다
7. 시작 버튼을 누른다 - 에이전트들이 하트비트를 시작하고 회사가 운영된다

## 가이드라인

Paperclip이 지원해야 하는 두 가지 런타임 모드가 있다:

- `local_trusted` (기본값): 로그인 마찰 없는 단일 사용자 로컬 신뢰 배포
- `authenticated`: 사설 네트워크 및 공개 배포 노출 정책 모두를 지원하는 로그인 필수 모드

표준 모드 설계와 명령 기대사항은 `doc/DEPLOYMENT-MODES.md`에 있다.
