# Paperclip 명세

Paperclip 컨트롤 플레인의 목표 명세. 살아있는 문서 - 명세 인터뷰 중 점진적으로 업데이트된다.

---

## 1. 회사 모델 (Company Model) [DRAFT]

회사(Company)는 일급 객체이다. 하나의 Paperclip 인스턴스가 여러 회사를 운영한다. 회사에는 독립된 "목표(goal)" 필드가 없다 - 방향은 이니셔티브(Initiative) 집합으로 정의된다 (태스크 계층 매핑 참조).

### 필드 (초안)

| Field       | Type          | Notes                             |
| ----------- | ------------- | --------------------------------- |
| `id`        | uuid          | Primary key                       |
| `name`      | string        | 회사 이름                          |
| `createdAt` | timestamp     |                                   |
| `updatedAt` | timestamp     |                                   |

### 이사회 거버넌스 (Board Governance) [DRAFT]

모든 회사에는 고영향 의사결정을 관장하는 **이사회(Board)**가 있다. 이사회는 인간 감독 계층이다.

**V1: 단일 인간 이사회.** 한 명의 인간 운영자.

#### 이사회 승인 게이트 (V1)

- 새 에이전트 고용 (새 에이전트 생성)
- CEO의 초기 전략적 분해 (CEO가 제안하고, 실행 시작 전 이사회가 승인)
- [TBD: 다른 거버넌스 게이트 액션 - 목표 변경, 에이전트 해고?]

#### 이사회 권한 (항상 사용 가능)

이사회는 전체 시스템에 대해 **무제한 접근 권한**을 항상 보유한다:

- **회사 예산 설정 및 수정** - 이사회가 최상위 토큰/LLM 비용 예산을 설정
- **에이전트 일시정지/재개** - 에이전트의 하트비트를 즉시 중지
- **작업 항목 일시정지/재개** - 태스크, 프로젝트, 하위 태스크 트리, 마일스톤을 일시정지. 일시정지된 항목은 에이전트에 의해 선택되지 않음.
- **전체 프로젝트 관리 접근** - UI를 통해 모든 태스크/프로젝트/마일스톤을 생성, 편집, 댓글, 수정, 삭제, 재배정
- **에이전트 결정 오버라이드** - 태스크 재배정, 우선순위 변경, 설명 수정
- **모든 레벨의 예산 수동 변경**

이사회는 단순한 승인 게이트가 아니다 - 실시간 제어 표면이다. 인간은 어느 레벨에서든 언제든지 개입할 수 있다.

#### 예산 위임 (Budget Delegation)

이사회가 회사 수준 예산을 설정한다. CEO는 하위 에이전트에 대한 예산을 설정할 수 있고, 모든 매니저 에이전트도 자신의 직속 보고자에 대해 동일하게 할 수 있다. 이 계단식 예산 위임이 실제로 어떻게 작동하는지는 TBD이지만, 권한 구조는 이를 지원한다. 이사회는 모든 레벨의 예산을 수동으로 오버라이드할 수 있다.

**향후 거버넌스 모델** (V1 아님):

- 고용 예산 (월 $X 이내 고용 자동 승인)
- 다중 구성원 이사회
- 위임된 권한 (CEO가 제한 내에서 고용 가능)

### 미결 사항 (Open Questions)

- 외부 수익/비용 추적 - 향후 플러그인. 토큰/LLM 비용 예산 관리가 핵심.
- 회사 수준 설정과 구성?
- 회사 생명주기 (일시정지, 아카이브, 삭제)?
- 고용과 CEO 전략 승인 외에 어떤 거버넌스 게이트 액션이 있는가?

## 2. 에이전트 모델 (Agent Model) [DRAFT]

모든 직원은 에이전트이다. 에이전트는 인력이다.

### 에이전트 정체성 (어댑터 수준)

SOUL.md (정체성/미션)와 HEARTBEAT.md (루프 정의) 같은 개념은 **Paperclip 프로토콜의 일부가 아니다**. 이들은 어댑터별 설정이다. 예를 들어, OpenClaw 어댑터는 SOUL.md와 HEARTBEAT.md 파일을 사용할 수 있다. Claude Code 어댑터는 CLAUDE.md를 사용할 수 있다. 순수 Python 스크립트는 커맨드라인 인자를 사용할 수 있다.

Paperclip은 에이전트가 정체성이나 행동을 정의하는 방법을 규정하지 않는다. 컨트롤 플레인을 제공하고, 어댑터가 에이전트의 내부 작동을 정의한다.

### 에이전트 설정 (Agent Configuration) [DRAFT]

각 에이전트는 **어댑터 타입**과 **어댑터별 설정 블롭(blob)**을 갖는다. 어댑터가 어떤 설정 필드가 존재하는지를 정의한다.

#### Paperclip 프로토콜 (Paperclip이 아는 것)

프로토콜 수준에서 Paperclip이 추적하는 것:

- 에이전트 정체성 (id, name, role, title)
- 조직 내 위치 (누구에게 보고하는지, 누가 보고하는지)
- 어댑터 타입 + 어댑터 설정
- 상태 (active, paused, terminated)
- 비용 추적 데이터 (에이전트가 보고하는 경우)

#### 어댑터 설정 (에이전트별)

각 어댑터 타입은 자체 설정 스키마를 정의한다. 예시:

- **OpenClaw adapter**: SOUL.md 내용, HEARTBEAT.md 내용, OpenClaw 전용 설정
- **Process adapter**: 실행할 명령, 환경 변수, 작업 디렉토리
- **HTTP adapter**: 엔드포인트 URL, 인증 헤더, 페이로드 템플릿

#### 내보내기 가능한 조직 설정 (Exportable Org Configs)

핵심 목표: **전체 조직의 에이전트 설정이 내보내기 가능해야 한다.** 회사의 전체 에이전트 설정 - 모든 에이전트, 어댑터 설정, 조직 구조 - 을 이식 가능한 아티팩트로 내보낼 수 있다. 이를 통해 다음이 가능하다:

- 회사 템플릿 공유 ("미리 구축된 마케팅 에이전시 조직이 있습니다")
- 회사 설정의 버전 관리
- 회사 복제/포크

#### 컨텍스트 전달 (Context Delivery)

에이전트별로 설정 가능하다. 스펙트럼의 양 끝:

- **Fat payload** - Paperclip이 관련 컨텍스트 (현재 태스크, 메시지, 회사 상태, 지표)를 하트비트 호출에 번들링한다. Paperclip에 콜백할 수 없는 단순/상태 비저장 에이전트에 적합.
- **Thin ping** - 하트비트는 단순한 깨우기 신호이다. 에이전트가 Paperclip API를 호출하여 필요한 컨텍스트를 가져온다. 자체 상태를 관리하는 고급 에이전트에 적합.

#### 최소 계약 (Minimum Contract)

Paperclip 에이전트가 되기 위한 최소 요구사항: **호출 가능할 것.** 그것뿐이다. Paperclip은 명령 또는 webhook으로 호출할 수 있다. 상태 보고 의무 없음 - Paperclip은 가능한 경우 프로세스 활성 상태로 기본 상태를 추론한다.

#### 통합 수준 (Integration Levels)

최소 요구사항 이상으로 Paperclip은 점진적으로 풍부한 통합을 제공한다:

1. **Callable** (최소) - Paperclip이 시작할 수 있다. 유일한 계약.
2. **Status reporting** - 에이전트가 실행 후 성공/실패/진행중을 보고한다.
3. **Fully instrumented** - 에이전트가 상태, 비용/토큰 사용량, 태스크 업데이트, 로그를 보고한다. 컨트롤 플레인과의 양방향 통합.

Paperclip은 전체 통합을 보여주는 **기본 에이전트**를 제공한다: 진행 추적, 비용 계측, 태스크 관리를 위한 **Paperclip skill** (Paperclip API와 상호작용하는 Claude Code 스킬). 이들은 유용한 기본값이자 어댑터 작성자를 위한 참조 구현으로 기능한다.

#### 내보내기 형식 (Export Formats)

두 가지 내보내기 모드:

1. **Template export** (기본) - 구조만: 에이전트 정의, 조직도, 어댑터 설정, 역할 설명. 선택적으로 시작을 돕는 몇 개의 시드 태스크를 포함. 새 회사를 만들기 위한 청사진.
2. **Snapshot export** - 전체 상태: 구조 + 현재 태스크, 진행 상황, 에이전트 상태. 복원하거나 포크할 수 있는 완전한 스냅샷.

일반적인 워크플로우: 템플릿을 내보내고, 이를 기반으로 새 회사를 만들고, 초기 태스크 몇 개를 추가하고, 시작한다.

## 3. 조직 구조 (Org Structure) [DRAFT]

계층적 보고 구조. CEO가 최상위이고, 보고가 아래로 전파된다.

### 에이전트 가시성 (Agent Visibility)

**조직 전체에 대한 완전한 가시성.** 모든 에이전트는 전체 조직도, 모든 태스크, 모든 에이전트를 볼 수 있다. 조직 구조는 **보고 및 위임 라인**을 정의하는 것이지, 접근 제어가 아니다.

각 에이전트는 자신의 책임과 역량에 대한 짧은 설명을 게시한다 - 스킬과 유사하다 ("내가 관련될 때"). 이를 통해 다른 에이전트가 누가 무엇을 도울 수 있는지 발견할 수 있다.

### 교차 팀 작업 (Cross-Team Work)

에이전트는 자신의 보고 라인 밖의 에이전트에게 태스크를 생성하고 배정할 수 있다. 이것이 교차 팀 협업 메커니즘이다. 이 규칙들은 주로 모든 에이전트에 권장되는 Paperclip SKILL.md에 인코딩되어 있다. Paperclip 앱은 도구와 약간의 거버넌스를 강제하지만, 아래의 교차 팀 규칙은 주로 에이전트 결정에 의해 구현된다.

#### 태스크 수락 규칙

에이전트가 팀 외부로부터 태스크를 받았을 때:

1. **적절하다고 동의 + 할 수 있음** → 직접 완료
2. **적절하다고 동의 + 할 수 없음** → 차단(blocked)으로 표시
3. **할 가치가 있는지 의문** → **스스로 취소할 수 없다.** 자신의 매니저에게 재배정하고 상황을 설명해야 한다. 매니저가 수락, 재배정, 또는 에스컬레이션 여부를 결정한다.

#### 매니저 에스컬레이션 프로토콜

부하 직원이 왜 차단되었는지 이해하고 해결하는 것은 모든 매니저의 책임이다:

0. **결정** - 매니저로서, 이 작업이 할 가치가 있는가?
1. **하향 위임** - 하위 에이전트에게 차단 해제를 요청
2. **상향 에스컬레이션** - 상위 매니저에게 도움 요청

#### 요청 깊이 추적 (Request Depth Tracking)

태스크가 교차 팀 요청에서 발생한 경우, **깊이(depth)**를 정수로 추적한다 - 원래 요청자로부터 몇 번의 위임 홉이 있었는지. 이를 통해 작업이 조직을 통해 얼마나 멀리 전파되는지 가시성을 제공한다.

#### 빌링 코드 (Billing Codes)

태스크에는 **빌링 코드**가 있어서 실행 중 토큰 지출을 요청 태스크/에이전트 상위로 귀속시킬 수 있다. 에이전트 A가 에이전트 B에게 작업을 요청하면, B 작업의 비용이 A의 요청에 대해 추적된다. 이를 통해 조직 전체에 걸친 비용 귀속이 가능해진다.

### 미결 사항 (Open Questions)

- 엄격한 트리인가, 에이전트가 여러 매니저에게 보고할 수 있는가?
- 런타임에 조직 구조가 변경될 수 있는가? (에이전트 재배정, 팀 재구성)
- 에이전트가 매니저로부터 설정을 상속하는가?
- 빌링 코드 형식 - 단순 문자열? 계층적?

## 4. 하트비트 시스템 (Heartbeat System) [DRAFT]

하트비트는 프로토콜이지 런타임이 아니다. Paperclip은 에이전트의 사이클을 시작하는 방법을 정의한다. 에이전트가 그 사이클에서 무엇을 하는지 - 얼마나 오래 실행하는지, 태스크 범위인지 연속적인지 - 는 전적으로 에이전트에 달려 있다.

### 실행 어댑터 (Execution Adapters)

에이전트 설정에는 Paperclip이 에이전트를 호출하는 방법을 정의하는 **어댑터**가 포함된다. 초기 어댑터:

| Adapter   | Mechanism               | Example                                       |
| --------- | ----------------------- | --------------------------------------------- |
| `process` | Execute a child process | `python run_agent.py --agent-id {id}`         |
| `http`    | Send an HTTP request    | `POST https://openclaw.example.com/hook/{id}` |

`process`와 `http` 어댑터가 기본으로 제공된다. 추가 어댑터는 플러그인 시스템을 통해 추가할 수 있다 (Plugin / Extension Architecture 참조).

### 어댑터 인터페이스 (Adapter Interface)

모든 어댑터는 세 가지 메서드를 구현한다:

```
invoke(agentConfig, context?) → void     // Start the agent's cycle
status(agentConfig) → AgentStatus        // Is it running? finished? errored?
cancel(agentConfig) → void               // Graceful stop signal (for pause/resume)
```

이것이 전체 어댑터 계약이다. `invoke`는 에이전트를 시작하고, `status`는 Paperclip이 상태를 확인하게 하고, `cancel`은 이사회의 일시정지 기능을 가능하게 한다. 나머지 모든 것 (비용 보고, 태스크 업데이트)은 선택 사항이며 Paperclip REST API를 통해 흐른다.

### Paperclip이 제어하는 것

- **언제** 하트비트를 발생시킬지 (스케줄/빈도, 에이전트별)
- **어떻게** 발생시킬지 (어댑터 선택 + 설정)
- **어떤 컨텍스트**를 포함할지 (thin ping vs. fat payload, 에이전트별)

### Paperclip이 제어하지 않는 것

- 에이전트가 얼마나 오래 실행하는지
- 에이전트가 사이클 중에 무엇을 하는지
- 에이전트가 태스크 범위인지, 시간 윈도우인지, 연속적인지

### 일시정지 동작 (Pause Behavior)

이사회 (또는 시스템)가 에이전트를 일시정지할 때:

1. **현재 실행에 신호** - 실행 중인 프로세스/세션에 정상 종료 신호를 보냄
2. **유예 기간** - 에이전트가 마무리하고, 상태를 저장하고, 최종 상태를 보고할 시간을 줌
3. **타임아웃 후 강제 종료** - 유예 기간 내에 에이전트가 멈추지 않으면 종료
4. **향후 하트비트 중지** - 에이전트가 재개될 때까지 새로운 하트비트 사이클이 발생하지 않음

이것은 "정상 신호 + 향후 하트비트 중지"이다. 현재 실행은 깔끔하게 착지할 기회를 갖는다.

### 미결 사항 (Open Questions)

- 하트비트 빈도 - 누가 제어하는가? 고정? 에이전트별? Cron 형태?
- 하트비트 호출이 실패하면 어떻게 되는가? (프로세스 충돌, HTTP 500)
- 건강 모니터링 - Paperclip이 "멈춤"과 "긴 태스크 작업 중"을 어떻게 구분하는가?
- 에이전트가 다음 하트비트를 자체 트리거할 수 있는가? ("끝났으니 5분 후에 다시 깨워줘")
- 유예 기간 - 고정? 에이전트별로 설정 가능?

## 5. 에이전트 간 통신 (Inter-Agent Communication) [DRAFT]

모든 에이전트 통신은 **태스크 시스템**을 통해 흐른다.

### 모델: 태스크 + 댓글

- **위임(delegation)** = 태스크를 생성하여 다른 에이전트에 배정
- **조율(coordination)** = 태스크에 댓글 달기
- **상태 업데이트** = 태스크 상태 및 필드 업데이트

별도의 메시징이나 채팅 시스템은 없다. 태스크가 통신 채널이다. 이를 통해 모든 컨텍스트가 관련 작업에 첨부되고, 자연스러운 감사 추적(audit trail)이 생성된다.

### 함의

- 에이전트의 "받은 편지함"은: 자신에게 배정된 태스크 + 참여 중인 태스크의 댓글
- CEO는 CTO에게 배정된 태스크를 생성하여 위임한다
- CTO는 이를 엔지니어에게 배정된 하위 태스크로 분해한다
- 논의는 태스크 댓글에서 이루어지며, 별도 채널이 아니다
- 에이전트가 에스컬레이션해야 하면, 상위 태스크에 댓글을 달거나 재배정한다

### 태스크 계층 매핑 (Task Hierarchy Mapping)

전체 계층: **이니셔티브(Initiative)** (회사 목표) → 프로젝트(Projects) → 마일스톤(Milestones) → 이슈(Issues) → 하위 이슈(Sub-issues). 모든 것은 이니셔티브로 거슬러 올라가며, "회사 목표"는 첫 번째/주요 이니셔티브일 뿐이다.

## 6. 비용 추적 (Cost Tracking) [DRAFT]

토큰/LLM 비용 예산 관리는 Paperclip의 핵심 기능이다. 외부 수익 및 비용 추적은 향후 플러그인이다.

### 비용 보고 (Cost Reporting)

완전 계측된 에이전트는 토큰/API 사용량을 Paperclip에 보고한다. 비용은 모든 수준에서 추적된다:

- **에이전트별** - 이 직원에게 얼마나 비용이 드는가?
- **태스크별** - 이 작업 단위에 얼마나 비용이 들었는가?
- **프로젝트별** - 이 산출물에 얼마나 비용이 드는가?
- **회사별** - 총 소진율(burn rate)

비용은 **토큰과 달러** 모두로 표기되어야 한다.

태스크의 빌링 코드 (조직 구조 참조)는 팀 간 비용 귀속을 가능하게 한다 - 에이전트 A가 에이전트 B에게 작업을 요청하면, B의 비용이 A의 요청으로 롤업된다.

### 예산 통제 (Budget Controls)

세 가지 단계:

1. **가시성(visibility)** - 모든 수준 (에이전트, 태스크, 프로젝트, 회사)에서 지출을 보여주는 대시보드
2. **소프트 알림(soft alerts)** - 설정 가능한 임계값 (예: 예산의 80%에서 경고)
3. **하드 한도(hard ceiling)** - 예산에 도달하면 에이전트 자동 일시정지. 이사회에 알림. 이사회가 오버라이드/한도 상향 가능.

예산은 **무제한(unlimited)**(한도 없음)으로 설정할 수 있다.

### 미결 사항 (Open Questions)

- 비용 보고 API - 에이전트가 비용을 보고하기 위한 스키마는?
- 대시보드 설계 - 각 수준에서 가장 중요한 지표는?
- 예산 기간 - 일별? 주별? 월별? 롤링?

## 7. 기본 에이전트 및 부트스트랩 플로우 (Default Agents & Bootstrap Flow) [DRAFT]

### 부트스트랩 시퀀스 (Bootstrap Sequence)

회사가 "생성됨"에서 "운영 중"으로 전환되는 과정:

1. 인간이 회사와 초기 이니셔티브를 생성
2. 인간이 초기 최상위 태스크를 정의
3. 인간이 CEO 에이전트를 생성 (기본 CEO 템플릿 또는 커스텀 사용)
4. CEO의 첫 하트비트: 이니셔티브와 태스크를 검토하고, 전략적 분해를 제안 (조직 구조, 하위 태스크, 고용 계획)
5. **이사회가 CEO의 전략 계획을 승인**
6. CEO가 실행을 시작 - 태스크 생성, 고용 제안 (이사회 승인), 위임

### 기본 에이전트 (Default Agents)

Paperclip은 기본 에이전트 템플릿을 제공한다:

- **Default Agent** - 기본 Claude Code 또는 Codex 루프. **Paperclip Skill** (SKILL.md)을 알고 있어 태스크 시스템과 상호작용하고, 회사 컨텍스트를 읽고, 상태를 보고할 수 있다.
- **Default CEO** - Default Agent를 CEO 전용 행동으로 확장: 전략 기획, 직속 보고자에 대한 위임, 진행 검토, 이사회 커뮤니케이션.

이들은 출발점이다. 사용자는 완전히 커스터마이즈하거나 교체할 수 있다.

### 기본 에이전트 동작 (Default Agent Behavior)

기본 에이전트의 루프는 **설정 기반(config-driven)**이다. 어댑터 설정에 에이전트가 각 하트비트 사이클에서 무엇을 하는지 정의하는 지시사항이 포함된다. 하드코딩된 표준 루프는 없다 - 각 에이전트의 설정이 행동을 결정한다.

이는 기본 CEO 설정이 CEO에게 전략을 검토하고, 직속 보고자를 확인하는 등의 지시를 한다는 의미이다. 기본 엔지니어 설정은 엔지니어에게 배정된 태스크를 확인하고, 가장 높은 우선순위를 선택해서 작업하라고 지시한다. 하지만 이들은 설정 선택이지 프로토콜 요구사항이 아니다.

### Paperclip Skill (SKILL.md)

에이전트에게 Paperclip과 상호작용하는 방법을 가르치는 스킬 정의. 다음을 제공한다:

- 태스크 CRUD (태스크 생성, 조회, 업데이트, 완료)
- 상태 보고 (체크인, 진행 보고)
- 회사 컨텍스트 (목표, 조직도, 현재 상태 읽기)
- 비용 보고 (토큰/API 사용량 로깅)
- 에이전트 간 통신 규칙

이 스킬은 어댑터에 무관하다 - Claude Code에 로드하거나, 프롬프트에 주입하거나, 커스텀 에이전트를 위한 API 문서로 사용할 수 있다.

## 8. 아키텍처 및 배포 (Architecture & Deployment) [DRAFT]

### 배포 모델 (Deployment Model)

**싱글 테넌트, 셀프 호스팅 가능.** SaaS가 아니다. 하나의 인스턴스 = 한 운영자의 회사들.

#### 개발 경로 (점진적 배포)

1. **Local dev** - 한 명령으로 설치하고 실행. 임베디드 Postgres. 모든 것이 로컬 머신에서. 에이전트가 로컬에서 실행.
2. **Hosted** - Vercel/Supabase/AWS/어디든 배포. 원격 에이전트가 공유 데이터베이스를 가진 서버에 연결. UI가 웹을 통해 접근 가능.
3. **Open company** - 선택적으로 일부를 공개 (예: 공개 회사를 위한 공개 채용 게시판).

핵심 제약: "내 머신에서 시도 중"에서 "내 에이전트들이 원격 서버에서 내 Paperclip 인스턴스와 통신하며 실행 중"으로의 전환이 쉬워야 한다.

#### 에이전트 인증 (Agent Authentication)

사용자가 에이전트를 생성하면, Paperclip이 서버 URL, API 키, 인증 방법에 대한 지시사항이 포함된 **연결 문자열(connection string)**을 생성한다. 에이전트는 그곳에서 토큰/키로 API를 호출하는 방법을 스스로 파악할 수 있다고 가정한다.

흐름:

1. 인간이 UI에서 에이전트를 생성
2. Paperclip이 연결 문자열 (URL + 키 + 지시사항)을 생성
3. 인간이 이 문자열을 에이전트에 제공 (예: 어댑터 설정, 환경 변수 등에)
4. 에이전트가 키를 사용하여 컨트롤 플레인에 대한 API 호출을 인증

### 기술 스택 (Tech Stack)

| Layer    | Technology                                                   |
| -------- | ------------------------------------------------------------ |
| Frontend | React + Vite                                                 |
| Backend  | TypeScript + Hono (REST API, tRPC 아님 - 비 TS 클라이언트 필요) |
| Database | PostgreSQL (`doc/DATABASE.md` 참조 - 개발용 PGlite 임베디드, 프로덕션용 Docker 또는 호스팅 Supabase) |
| Auth     | [Better Auth](https://www.better-auth.com/)                  |

### 동시성 모델: 원자적 태스크 체크아웃 (Atomic Task Checkout)

태스크는 **단일 배정** (태스크당 에이전트 한 명)과 **원자적 체크아웃**을 사용한다:

1. 에이전트가 태스크를 `in_progress`로 설정 시도 (소유권 주장)
2. API/데이터베이스가 이를 원자적으로 강제 - 다른 에이전트가 이미 소유권을 주장했으면, 어떤 에이전트가 갖고 있는지 식별하는 오류와 함께 요청이 실패
3. 태스크가 이전 세션에서 요청 에이전트에 이미 배정되어 있으면, 재개 가능

낙관적 잠금이나 CRDT 불필요. 단일 배정 모델 + 원자적 체크아웃이 설계 수준에서 충돌을 방지한다.

### 인간 인더 루프 (Human in the Loop)

에이전트는 인간에게 배정된 태스크를 생성할 수 있다. 이사회 구성원 (또는 접근 권한이 있는 인간)이 UI를 통해 이러한 태스크를 완료할 수 있다.

인간이 태스크를 완료하면, 요청 에이전트의 어댑터가 **핑백(pingbacks)**을 지원하는 경우 (예: OpenClaw 훅), Paperclip이 해당 에이전트를 깨우기 위한 알림을 보낸다. 이를 통해 인간은 드물지만 가능한 워크플로우 참여자가 된다.

에이전트는 Paperclip SKILL에서 인간에게 태스크를 배정하지 않도록 권장되지만, 때로는 불가피하다.

### API 설계 (API Design)

**단일 통합 REST API.** 동일한 API가 프론트엔드 UI와 에이전트 모두를 서빙한다. 인증이 권한을 결정한다 - 이사회 인증은 전체 접근, 에이전트 API 키는 범위 제한 접근 (자신의 태스크, 비용 보고, 회사 컨텍스트).

별도의 "에이전트 API" vs. "이사회 API"가 아니다. 동일한 엔드포인트, 다른 인가 수준.

### 작업 산출물 (Work Artifacts)

Paperclip은 작업 산출물 (코드 저장소, 파일 시스템, 배포, 문서)을 **관리하지 않는다**. 그것은 전적으로 에이전트의 영역이다. Paperclip은 태스크와 비용을 추적한다. 작업이 어디서 어떻게 이루어지는지는 범위 밖이다.

### 미결 사항 (Open Questions)

- UI로의 실시간 업데이트 - WebSocket? SSE? Polling?
- 에이전트 API 키 범위 - 에이전트가 정확히 무엇에 접근할 수 있는가? 자신의 태스크만? 팀의? 전체 회사?

### 충돌 복구: 수동, 자동 아님 (Crash Recovery: Manual, Not Automatic)

에이전트가 태스크 중간에 충돌하거나 사라지면, Paperclip은 태스크를 자동 재배정하거나 자동 해제하지 **않는다**. 대신:

- Paperclip이 오래된 태스크 (최근 활동 없이 `in_progress` 상태인 태스크)를 대시보드와 보고서를 통해 표면화
- Paperclip은 조용히 실패하지 않는다 - 감사 및 가시성 도구가 문제를 명백하게 만듦
- 복구는 인간 또는 창발적 프로세스에 의해 처리 (예: 오래된 작업을 모니터링하고 표면화하는 것이 임무인 프로젝트 매니저 에이전트)

**원칙: Paperclip은 문제를 보고하지, 조용히 수정하지 않는다.** 자동 복구는 실패를 숨긴다. 좋은 가시성은 적절한 주체 (인간 또는 에이전트)가 무엇을 할지 결정하게 한다.

### 플러그인 / 확장 아키텍처 (Plugin / Extension Architecture)

핵심 Paperclip 시스템은 확장 가능해야 한다. 지식 기반, 외부 수익 추적, 새 에이전트 어댑터 같은 기능은 코어를 수정하지 않고 **플러그인**으로 추가할 수 있어야 한다. 이는 다음을 의미한다:

- 플러그인이 연결할 수 있는 잘 정의된 API 경계
- 태스크/에이전트 생명주기 이벤트에 반응하기 위한 이벤트 시스템 또는 훅
- **에이전트 어댑터 플러그인** - 새 어댑터 타입을 플러그인 시스템을 통해 등록 가능
- 플러그인 등록 가능 UI 컴포넌트 (향후)

이것은 V1 산출물이 아니다 (플러그인 프레임워크를 사전에 구축하지 않음). 하지만 아키텍처가 막다른 길로 이끌어서는 안 된다. 확장이 가능하도록 경계를 깨끗하게 유지한다.

## 9. 프론트엔드 / UI (Frontend / UI) [DRAFT]

### 주요 뷰 (Primary Views)

각각 별도의 페이지/라우트이다:

1. **조직도(Org Chart)** - 에이전트별 실시간 상태 표시기 (running/idle/paused/error)가 있는 조직 트리. 에이전트가 무엇을 하고 있는지의 실시간 활동 피드.
2. **태스크 보드(Task Board)** - 태스크 관리. 칸반과 리스트 뷰. 팀, 에이전트, 프로젝트, 상태별 필터.
3. **대시보드(Dashboard)** - 고수준 지표: 에이전트 수, 활성 태스크, 비용, 목표 진행률, 소진율. GOAL.md의 "한눈에 보기" 뷰.
4. **에이전트 상세(Agent Detail)** - 단일 에이전트 심층 분석: 태스크, 활동, 비용, 설정, 상태 이력.
5. **프로젝트/이니셔티브 뷰(Project/Initiative Views)** - 마일스톤과 목표에 대한 진행 추적.
6. **비용 대시보드(Cost Dashboard)** - 모든 수준 (에이전트, 태스크, 프로젝트, 회사)에서의 지출 시각화.

### 이사회 제어 (모든 뷰에서 사용 가능)

- 에이전트 일시정지/재개 (모든 뷰)
- 태스크/프로젝트 일시정지/재개 (모든 뷰)
- 보류 중인 액션 승인/거부 (고용, 전략 제안)
- 직접 태스크 생성, 편집, 댓글

## 11. 지식 기반 (Knowledge Base)

**핵심에 대한 안티 목표.** 지식 기반은 Paperclip 코어의 일부가 아니다 - 플러그인이 될 것이다. 태스크 시스템 + 댓글 + 에이전트 설명이 충분한 공유 컨텍스트를 제공한다.

아키텍처는 나중에 지식 기반 플러그인을 추가하는 것을 지원해야 하지만 (깨끗한 API 경계, 연결 가능한 생명주기 이벤트), 핵심 시스템은 명시적으로 이를 포함하지 않는다.

---

## 12. 안티 요구사항 (Anti-Requirements)

Paperclip이 명시적으로 하지 **않는** 것들:

- **에이전트 런타임이 아니다** - Paperclip은 오케스트레이션하고, 에이전트는 다른 곳에서 실행
- **지식 기반이 아니다** - 코어에 위키/문서/벡터 DB 없음 (플러그인 영역)
- **SaaS가 아니다** - 싱글 테넌트, 셀프 호스팅
- **에이전트 구현에 대해 의견을 갖지 않는다** - 어떤 언어, 프레임워크, 런타임이든
- **자동 자가 치유하지 않는다** - 문제를 표면화하지, 조용히 수정하지 않음
- **작업 산출물을 관리하지 않는다** - 저장소 관리, 배포, 파일 시스템 없음
- **작업을 자동 재배정하지 않는다** - 오래된 태스크는 표면화되지, 조용히 재분배되지 않음
- **외부 수익/비용을 추적하지 않는다** - 향후 플러그인. 토큰/LLM 비용 예산 관리가 핵심.

## 13. 통합 원칙 (Principles, Consolidated)

1. **에이전트 실행 방식에 무관하다.** 어떤 언어, 프레임워크, 런타임이든. Paperclip은 컨트롤 플레인이지 실행 플레인이 아니다.
2. **회사가 조직의 단위이다.** 모든 것은 회사 아래에 존재한다.
3. **태스크가 통신 채널이다.** 모든 에이전트 통신은 태스크 + 댓글을 통해 흐른다. 별도 채널 없음.
4. **모든 업무는 목표로 이어진다.** 계층적 태스크 관리 - 어떤 것도 고립되어 존재하지 않는다.
5. **이사회가 통치한다.** 인간이 이사회를 통해 통제를 유지한다. 보수적 기본값 (인간 승인 필요).
6. **문제를 표면화하되, 숨기지 않는다.** 좋은 감사와 가시성. 조용한 자동 복구 없음.
7. **원자적 소유권.** 태스크당 단일 담당자. 원자적 체크아웃이 충돌을 방지.
8. **점진적 배포.** 로컬에서 시작하기 쉽고, 호스팅으로 확장하기 간단.
9. **확장 가능한 코어.** 플러그인이 코어 수정 없이 기능 (어댑터, 지식 기반, 수익 추적)을 추가할 수 있도록 깨끗한 경계 유지.
